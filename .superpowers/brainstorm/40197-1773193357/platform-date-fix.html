Platform Date Handling — Problem vs. Fix

Example: You select "Last 30 days" → Feb 10 – Mar 12, 2026

MAVELY Transaction-level
❌ CURRENT
Returns full calendar month aggregates from mavely_links. "30 days" returns all of Feb + all of Mar regardless of your actual date range.
✅ FIXED
Query mavely_transactions directly: WHERE sale_date BETWEEN Feb 10 AND Mar 12. Returns exact revenue for exactly those 30 days. Day-accurate.
Displays as: $2,341 · Feb 10 – Mar 12
SHOPMY Transaction-level
❌ CURRENT
Fetches last 100 commissions from API, then filters in JavaScript. If there are 150+ transactions in range, older ones are silently missing. No date param sent to API.
✅ FIXED
Query already-synced sales table in DB: WHERE sale_date BETWEEN Feb 10 AND Mar 12. No API call at query time. Synced data is complete in DB.
Displays as: $891 · Feb 10 – Mar 12
LTK Monthly only
❌ CURRENT
Returns whatever the last sync returned — usually just the current calendar month with no date filtering applied at all. Selecting "last 30 days" changes nothing.
✅ FIXED
Query platform_earnings for months that overlap the selected range. Feb 10–Mar 12 → return Feb 2026 bucket + Mar 2026 bucket. Label them clearly — no false precision.
Displays as:
Feb 2026: $3,100  |  Mar 2026: $1,240 (so far)
⚠ LTK reports by month — showing full months that overlap your range
IMPACT.COM AMAZON Monthly only
❌ CURRENT
Same as LTK — no date filtering on the API query. Returns last synced data regardless of selected range.
✅ FIXED
Same fix as LTK — query platform_earnings by overlapping months. Show full month buckets with the same "monthly reporting" label. Consistent pattern across all monthly platforms.
Displays as:
Feb 2026: $540  |  Mar 2026: $190 (so far)
⚠ Amazon reports by month — showing full months that overlap your range
