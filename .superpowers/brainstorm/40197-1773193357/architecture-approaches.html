How should global filters (creator + date) be wired?

All 3 options produce the same UX — the difference is how the state lives and flows through the app.

A

URL Params as Source of Truth RECOMMENDED

Filters live in the URL: ?creator=nicki&start=2026-02-10&end=2026-03-12

A useFilters() hook reads useSearchParams() and pushes updates to the router. Server components receive params directly as page props — no client state needed.

Data flow:
URL params → useFilters() hook → components + API calls
Filter change → router.push() → URL updates → page re-renders

Pros

  • Survives page refresh
  • Shareable/bookmarkable
  • No extra state management
  • Next.js App Router native pattern

Cons

  • Slightly more routing boilerplate
B

React Context (in-memory)

A FilterProvider wraps the app and holds creator + date range in React state. All pages and components read from useFilterContext().

Data flow:
FilterProvider → context → components + API calls
Filter change → setState → re-render (URL stays the same)

Pros

  • Simple to implement
  • No routing complexity

Cons

  • Lost on page refresh
  • Not shareable
  • Context re-renders can cascade
C

URL + Context Hybrid

URL is source of truth AND a context layer mirrors it for ergonomics. On mount, context hydrates from URL. On change, both URL and context update together.

Data flow:
URL → hydrate context on mount
Filter change → update URL + context simultaneously

Pros

  • Shareable URLs
  • Convenient context access

Cons

  • Two sources of truth to keep in sync
  • More complex, more bugs
  • Overkill for internal tool
